// 01 Overview

프로젝트 개요 및 배경

의료 현장의 정보 접근성 문제를 AI로 해결한 2개년 프로젝트

부산광역시 감염병관리지원단은 매년 전국 의료기관에 법정감염병 144종에 대한 핸드북을 발간합니다. 그러나 긴급한 대응이 요구되는 의료 현장에서 수백 페이지의 문서를 수동으로 검색하는 방식은 시간적 비효율과 정보 접근성 저해를 초래했습니다.

이 구조적 문제를 해결하기 위해 자연어 질의로 즉시 원하는 정보를 찾아주는 AI 챗봇 시스템의 개발이 요구되었으며, 발주처는 세 가지 핵심 요구사항을 제시했습니다.

#요구사항기술적 과제
1144종 법정감염병 최신 지침서 내용 100% 반영문서 전처리 파이프라인 + 벡터 DB 설계
2답변 시 공식 문서의 출처 URL 반드시 표기메타데이터 설계 및 Source 매핑 로직

// 03 IDBOT v1 (2024)

1차년도 — Naive RAG 구축

144종 질병 정보를 1,440개 HTML로 원자화한 Naive RAG 시스템

IDBOT v1 — Naive RAG Architecture (2024)

핵심 설계: 1,440개 HTML 원자화

144종 질병 정보를 10개 카테고리(질병 개요, 증상, 진단, 치료, 예방 등)로 분절하여 총 1,440개의 독립적인 HTML 문서로 원자화했습니다. 보건의료 전문가와의 협업을 통해 의료 현장에서 실제로 어떤 정보 단위로 검색이 이루어지는지를 반영한 설계입니다.

각 HTML 문서는 Netlify를 통해 개별 URL을 부여받아, 발주처의 핵심 요구사항인 '출처 URL 표기'를 기술적으로 구현했습니다.

// 04 IDBOT v2 (2025)

2차년도 — Advanced RAG 고도화

Pre-Retrieval → Retrieval → Post-Retrieval 3단계 파이프라인으로 전면 재설계

IDBOT v2 — Advanced RAG 3-Stage Pipeline Architecture (2025)

3단계 파이프라인 상세

1
Pre-Retrieval — 질문 전처리 및 라우팅
IntentService가 질문 유형(인사/챗봇정보/통계/질병질의)을 분류하여 불필요한 벡터 검색을 조기 차단합니다. NormalizationService는 142종 동의어 사전을 통해 "신종플루" → "신종인플루엔자 A(H1N1)"로 자동 변환하고, RoutingService가 키워드 분석을 통해 최적의 Vector Store를 결정합니다.
IntentService NormalizationService GradeService RoutingService
2
Retrieval — 12개 도메인별 FAISS 병렬 검색
RetrieverLoader가 라우팅 결과에 따라 Retriever를 동적 로드하고, asyncio.gather()를 통해 12개 도메인별 FAISS 인덱스를 병렬 비동기 검색합니다. 단일 인덱스 대비 검색 정확도를 높이면서 레이턴시를 2초 이내로 관리합니다.
RetrieverLoader FAISS × 12 asyncio.gather() bge-m3 Embedding
3
Post-Retrieval — 결과 병합, 답변 생성, 출처 태깅
AggregatorService가 복수 Retriever 결과를 병합하고 중복을 제거합니다. SummarizerService가 Ollama EXAONE 3.5 7.8B로 근거 기반 최종 답변을 생성하며, SourceService가 공식 문서 출처 URL을 자동 매핑하여 답변에 추가합니다.
AggregatorService SummarizerService SourceService EXAONE 3.5 7.8B
// 05 Data Pipeline

문서 전처리 파이프라인 혁신

LLM 호출 0회 — 완전한 재현성과 Zero Cost 전처리

핵심 설계 결정: 파이프라인 전체에서 GPT·Gemini 등 생성형 LLM을 사용하지 않았습니다. 문서 파싱은 Upstage Document Parse API(OCR)에, 구조화 로직은 순수 Python 정규식·문자열 파싱에 위임하여 비용과 레이턴시를 최소화하고 동일 입력에 대한 완전한 재현성을 보장했습니다.

Pipeline A: 관리 지침서 처리

카테고리별 관리 지침서 11종을 각각 독립적인 FAISS 인덱스로 구축합니다. Upstage API가 모든 제목을 # 단일 레벨로 반환하는 한계를 극복하기 위해, 정규식 패턴 매칭으로 H1/H2/H3 논리적 계층 구조를 재구축하는 헤딩 계층 승격(Heading Elevation) 기법을 적용했습니다.

# 헤딩 계층 승격 패턴
H1_PATTERNS = [re.compile(r"^제\s*\d+\s*장"), ...]
H2_PATTERNS = [re.compile(r"^\d+[\.\s]+\S"), ...]
H3_PATTERNS = [re.compile(r"^\d+\)\s"), ...]
# 결과: H1 2,426 + H2 2,740 + H3 4,106 생성

Pipeline B: 알아보기 문서 처리 (LangGraph)

약 1,400개의 PDF를 처리하기 위해 LangGraph 5노드 그래프를 설계했습니다. 핵심 후처리 로직인 convert_table_to_sections()는 Upstage API가 반환하는 2열 표를 섹션 구조로 변환하여 청킹 시 의미 단위가 완벽하게 보존되도록 했습니다.

MD5 해시 키 기반 캐싱 시스템을 구현하여, 1,400개 파일 처리 중 어느 시점에 중단되더라도 안전하게 재개할 수 있습니다.

# 표 → 섹션 변환
# 변환 전: | 감염병 분류 | ◦ 제1급 법정감염병 |

# 변환 후:
