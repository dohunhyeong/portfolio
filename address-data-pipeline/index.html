// 01. Problem
80만 건 처리를 가로막는 두 가지 장벽
2024년 부산대학교 병원에서 의뢰한 코로나19 환자 데이터의 주소 전처리 작업에서 두 가지 근본적인 문제에 직면했습니다.
⏱️
시간적 장벽
수동 웹 검색 방식으로는 일일 최대 처리량이 약 5,000건으로, 전체 80만 건을 처리하는 데 약 160일이 소요될 것으로 예상되었습니다. 프로젝트 마감 기한 내 완수가 물리적으로 불가능한 상황이었습니다.

// 04. Architecture
6단계 파이프라인 전체 구조
원본 Excel 파일에서 시작하여 완성 데이터까지 이어지는 6단계 파이프라인 전체 흐름입니다. 각 단계는 독립 스크립트로 구성되어 개별 실행 및 재시도가 가능합니다.
Architecture 1 — 6-Stage Pipeline
0
데이터 준비 — 원본 Excel → CSV 변환
preprocessing.ipynb
원본 Excel 파일의 인코딩을 UTF-8로 변환하고, 필요한 컬럼을 검증하여 CSV 형식으로 저장합니다.
1
데이터 분할 — 5,000건 단위 배치 생성
data_splitting.py
80만 건의 데이터를 5,000건 단위로 분할합니다. 네트워크 불안정 시 최대 손실 범위를 관리 가능한 수준으로 제한하고, 중간 결과물 저장으로 진행 상황 추적이 용이합니다.
2
1차 API 동기화 — Juso API 주소 변환
sync_address.py
각 배치에 대해 Juso API를 호출하여 주소를 표준화합니다. 성공: final_address_N.csv / 실패: error_address_N.csv
3
2차 에러 복구 — 수동 정제 후 재처리
sync_error.py
1차 실패 데이터를 수동으로 정제(통/반·상세주소 제거)한 후 2차 API 처리를 수행합니다. 원본 인덱스를 보존하여 정확한 위치에 결과를 병합합니다.
4
최종 병합 — 전체 배치 통합
merge.py & error_merge.py
pd.isna() 체크로 선택적 업데이트하여 기존 성공 데이터를 보호하고, 전체 배치를 final_merged_output.xlsx로 통합합니다.
5
수동 검토 — 최종 잔여 에러 처리
주소정보누리집 활용
자동화 이후에도 남은 NaN 값(폐지된 주소 등)을 주소정보누리집에서 수동으로 검색하여 100% 정확도를 달성합니다.
// 05. Technical Detail
지능형 API 재시도 로직: 3단계 에러 분류로 안정성 확보
단순 재시도가 아닌, 에러 유형을 3단계로 분류하여 각각에 맞는 처리 전략을 적용했습니다.
python · call_juso_api()
def call_juso_api(address, retries=3, delay=3):
    params = {
        "keyword": address,
        "confmKey": CONFIRM_KEY,
        "hstryYn": "Y",  # 폐지된 주소도 검색
        "resultType": "json",
    }
    for attempt in range(retries):
        try:
            response = requests.get(API_URL, params=params, timeout=50)
            if response.status_code == 200:
                data = response.json()
                if data["results"]["common"]["errorCode"] == "0" and data["results"]["juso"]:
                    juso = data["results"]["juso"][0]
                    return {"시/도": juso.get("siNm"), "시/군/구": juso.get("sggNm"), "읍/면/동": juso.get("emdNm")}
        except requests.exceptions.RequestException as e:
            logger.error(f"RequestException: {e}")
        if attempt < retries - 1:
            time.sleep(delay)  # 3초 대기 후 재시도
    return None
retries=3
일시적 네트워크 오류 대응. 3회 이상은 영구적 실패로 판단하여 에러 파일로 분리.
delay=3
API 서버 부하 방지 및 Rate Limit 회피. 재시도 전 3초 대기.
timeout=50
대용량 요청 시 응답 지연 고려. 50초 이상 무응답 시 에러로 처리.
hstryYn="Y"
폐지된 주소도 검색하여 변환 성공률 향상. 도로명 변경 이전 주소 처리 가능.
// 06. Technical Decision
5,000건 배치 전략: 처리 효율과 장애 복구의 최적 균형점
배치 크기는 처리 효율과 장애 복구 비용 사이의 트레이드오프를 고려하여 결정했습니다.
배치 크기처리 시간장점단점
1,000건 약 5~8분 빠른 실패 복구 파일 수 과다, I/O 오버헤드 증가
5,000건 ✓ 약 25~40분 균형잡힌 처리 단위, 진행 추적 용이 —
10,000건 약 50~80분 파일 수 감소 실패 시 재처리 비용 증가
50,000건+ 4시간+ — 네트워크 타임아웃 위험, 복구 불가 수준
// 07. Error Recovery
2단계 에러 복구 시스템: 자동+수동 조합으로 100% 달성
자동화만으로 처리 불가능한 케이스를 수동 정제와 조합하여 최종 100% 정확도를 달성했습니다. 아래 다이어그램은 에러 복구 전체 흐름을 나타냅니다.
Architecture 2 — 2-Stage Error Recovery System
설계 핵심 원칙
1. pd.isna() 체크 → 이미 성공한 데이터 덮어쓰기 방지 2. 원본 인덱스 보존 → 배치 분할 이후에도 정확한 위치에 결과 병합 3. 자동+수동 조합 → 100% 정확도 달성
Stage 2
1차 API 처리 — sync_address.py
전체 배치에 대해 Juso API를 호출합니다. 성공 결과는 final_address_N.csv로, 실패 결과는 error_address_N.csv로 분리 저장됩니다.
수동
수동 주소 정제 — 통/반·상세주소 제거
에러 파일을 열어 API 실패 원인을 분석하고, 상세주소·통/반 정보 등 불필요한 텍스트를 제거합니다.
Stage 3
2차 API 처리 — sync_error.py
수동 정제된 주소로 2차 API 처리를 수행합니다. 원본 Index 컬럼을 보존하여 최종 병합 시 정확한 위치에 결과를 삽입합니다.
병합
NaN 선택적 업데이트 — error_merge.py
pd.isna() 체크로 이미 성공한 데이터는 보호하고, NaN 값인 경우에만 복구 결과로 업데이트합니다.
Stage 5
최종 수동 검토 — 주소정보누리집
통합 후에도 남은 NaN 값을 확인합니다. 폐지된 주소는 주소정보누리집에서 "폐지된 주소 정보 포함" 옵션으로 검색하여 수동 입력합니다. 이 단계를 통해 최종 100% 정확도를 달성합니다.
