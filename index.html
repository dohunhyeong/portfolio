Problem & Solution

두 가지 장벽, 하나의 해결책

80만 건의 코로나19 환자 주소 데이터를 처리하는 데 있어 기술적·시간적 한계에 직면했습니다.

⏱️
시간적 장벽
수동 웹 검색 방식으로는 일일 최대 처리량이 약 5,000건으로, 전체 80만 건을 처리하는 데 약 160일이 소요될 것으로 예상되었습니다. 프로젝트 마감 기한 내 완수가 물리적으로 불가능한 상황이었습니다.
🔧
기술적 장벽
도로명 주소와 지번 주소의 체계 차이로 인해 단순 정규식만으로는 변환에 한계가 있었습니다. 상세주소, 통/반 정보, 폐지된 주소 등 다양한 비정형 패턴이 혼재했습니다.
🔍
해결책 발견: Juso API 분석
행정안전부의 Juso API를 분석하여 자동화 가능성을 확인했습니다. API 요청 실패 원인을 면밀히 파악하고, 정제 로직과 수동 개입이 조화된 2단계 에러 복구 시스템을 설계했습니다.
🚀
자동화 파이프라인 구축
6단계 파이프라인을 설계하여 비정형 주소를 표준 행정구역 체계(시/도, 시/군/구, 읍/면/동)로 자동 변환했습니다. 160일 → 1일로 처리 시간을 단축했습니다.
Architecture

6단계 파이프라인 구조

원본 Excel 파일에서 완성 데이터까지, 각 단계별 역할과 스크립트를 확인하세요.

0
데이터 준비 — 원본 Excel → CSV 변환
preprocessing.ipynb
원본 Excel 파일의 인코딩을 UTF-8로 변환하고, 필요한 컬럼을 검증하여 CSV 형식으로 저장합니다.
1
데이터 분할 — 5,000건 단위 배치 생성
data_splitting.py
80만 건의 데이터를 5,000건 단위로 분할합니다. 네트워크 불안정 시 최대 손실 범위를 관리 가능한 수준으로 제한합니다.
2
1차 API 동기화 — Juso API 주소 변환
sync_address.py
각 배치에 대해 Juso API를 호출하여 주소를 표준화합니다. 성공 결과는 final_address_N.csv, 실패 결과는 error_address_N.csv로 분리 저장됩니다.
3
2차 에러 복구 — 수동 정제 후 재처리
sync_error.py
1차 실패 데이터를 수동으로 정제(통/반·상세주소 제거)한 후, 2차 API 처리를 수행합니다. 원본 인덱스를 보존하여 정확한 위치에 결과를 병합합니다.
4
최종 병합 — 전체 배치 통합
merge.py & error_merge.py
복구 성공 데이터를 pd.isna() 체크로 선택적 업데이트하여 기존 성공 데이터를 보호하고, 전체 배치를 final_merged_output.xlsx로 통합합니다.
5
수동 검토 — 최종 잔여 에러 처리
주소정보누리집 활용
자동화 이후에도 남은 NaN 값(폐지된 주소 등)을 주소정보누리집에서 수동으로 검색하여 100% 정확도를 달성합니다.
Tech Stack

핵심 기술 및 설계 원칙

데이터 안정성과 처리 효율을 동시에 달성하기 위한 기술적 선택들입니다.

Python & Pandas
대용량 데이터를 배치 단위로 분할하여 메모리 초과 없이 안전하게 처리. pd.isna() 기반 선택적 업데이트로 데이터 무결성 보장.
Juso API (행정안전부)
공공 데이터 포털 주소 API 연동. hstryYn="Y" 옵션으로 폐지된 주소까지 검색하여 변환 성공률 향상.
지능형 재시도 로직
HTTP 에러, API 에러, 네트워크 에러를 3단계로 분류하여 처리. retries=3, delay=3초의 백오프 전략으로 API 서버 부하 방지.
배치 처리 아키텍처
5,000건 단위 분할로 API 호출당 약 25~40분 처리. 네트워크 불안정 시 최대 손실을 5,000건으로 제한하여 장애 복구 비용 최소화.
원본 인덱스 보존
배치 분할 이후에도 원본 Index 컬럼을 보존하여, 에러 복구 후 정확한 위치에 결과를 병합. 데이터 일관성 100% 유지.
자동+수동 하이브리드
자동화로 처리 불가능한 폐지 주소, 특수 케이스는 수동 검토로 보완. 두 방식의 조합으로 100% 정확도 달성.
